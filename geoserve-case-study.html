Business challenge

A trading team running on Bloomberg, WhatsApp, and manual spreadsheets

Maritime fuel trading is a time-critical, margin-thin business. When I joined Geoserve, the trading team managed their entire operation across multiple external tools with no single source of truth - prices were stale by the time they were referenced, deal pipeline visibility was non-existent, and critical follow-ups were scattered across email threads.

The opportunity: a unified intelligence platform purpose-built for the way traders actually work - live commodity pricing, global port bunkering rates, active deal status, and a real-time port map in one fast interface.

Discovery & research

Three user interviews, four design-defining insights

Before touching Figma, I ran structured interviews with the core users - a Senior Trader, a Trader, and a Sales lead. Each had a different workflow and different gaps in the existing system.

KT
Karthik
Senior Trader
Needed configurable KPIs - performance signals differ by seniority
Deal pattern analytics to understand which ports close fastest
Port-level map view to see pricing in geographic context
GeoServe-specific pricing separate from external market data
FL
Felicia
Trader
Relied on Sea Searcher for ship tracking - not in GeoStem at all
No visibility of customer credit line when selecting in inquiry - a risk gap
Traders closest to real market rates but couldn't enter price indications
Claims pending not surfaced in follow-ups, buried in emails
SF
Saif
Sales
Ran 4 separate external tools daily alongside GeoStem
Trade Winds for 5-min live pricing, Splash for news, Sea Searcher for ship locations
Wanted broader crude pricing options beyond just Brent and WTI
Performance dashboard valued but fuel-split data too granular for sales role

Four themes shaped the design direction:

01
Tool consolidation
Every user ran 3-5 external tools alongside GeoStem. Significant opportunity to consolidate if depth matched Sea Searcher and Trade Winds.
02
Role-based KPIs
Senior Traders, Traders, and Sales have fundamentally different performance signals. One dashboard view can't serve all three.
03
Trader write access
Price indications could only be entered by analysts, but traders were closest to real market data. A critical workflow gap.
04
Context-rich follow-ups
Follow-ups needed more than a task label - claims status, credit line, and inquiry history needed to be visible in-context.

Strategy

Two modules, one design system

Working with the CEO, I scoped the platform into two modules: the GeoStem trading dashboard (live pricing, port intelligence, deal pipeline) and a Voyage Optimisation Report system (weather routing, waypoint planning, delivered as a structured multi-page report). I built the design system in parallel with the first module - every component built for the dashboard became a reusable asset for voyage reporting.

Design process

From information hierarchy to shipped interface

01

Three-panel layout system

Ranked every data point by trader frequency and urgency. Live commodity price and active follow-ups topped the list. This became the layout architecture: left panel for market data and deal actions, centre for port pricing, right for trader updates and news. Each section independently scrollable - traders monitor all three contexts without switching views.

02

Live fuel price chart

A dedicated full-screen view with TradingView integration for commodity price history across Brent, Crude, and Natural Gas. Six stat cards (Open, High, Low, Prev Close, 52W High, 52W Low) with colour-coded deltas that read instantly without scanning labels.

03

Interactive global port map

Leaflet-based port map with a custom dark navy tile treatment. Clicking a port surfaces fuel grades (VLSFO, HSFO, LSMGO), weather conditions, top sellers, and nearby ports. Overlays panel for maritime context - ECA zones, piracy areas, storm routing, and base layer modes.

04

Built in Claude Code - shared with dev in 5 days

The entire UI was built in Claude Code as a fully interactive prototype - all screens, interaction states, data edge cases (empty states, loading, error, partial data), and responsive behaviour. Shared with the dev team in 5 days. This meant developers didn't write UI code from scratch - they connected their backend to a production-ready interface. Result: significantly faster go-live and fewer UI bugs in production.

Outcomes

What shipped and what it changed

GeoStem consolidated 3-4 external tools into one fast interface. Traders reported spending significantly less time locating information during active sessions, and deal pipeline visibility - previously non-existent - was now surfaced in a single view. The shared design system continues to pay dividends across voyage reporting, validation screens, and onboarding flows.
